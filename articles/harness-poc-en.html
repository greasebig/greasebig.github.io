Building a Harness from Scratch: ~300 Lines of Python

2026-05-24 · English reader version

This is the implementation follow-up to the Harness Engineering essay: a minimal but usable agent harness that proves the core loop.

Goal

The prototype validates the smallest useful set of ideas: multi-provider model calls, an agent loop, automatic context loading from AGENTS.md, persistent memory, permission boundaries, and lifecycle hooks.

Shape of the System

harness/
├── agent.py       # agent loop
├── context.py     # AGENTS.md discovery
├── memory.py      # persistent MEMORY.md
├── permissions.py # path and command checks
├── tools.py       # tool definitions
├── hooks.py       # PreToolUse / PostToolUse
└── cli.py         # command-line entry

Agent Loop

The loop is intentionally simple: send the user task and system context, stream the model response, execute tool calls if allowed, append tool results, and repeat until the model returns a final answer. Reliability comes from the boundary around the loop, not from making the loop complicated.

Important Lessons

  • Context should be short and navigational, not encyclopedic.
  • Memory should be explicit and durable, not accidental prompt clutter.
  • Permissions must be enforced by code, not described as a suggestion.
  • Hooks are where teams add policy, logging, metrics, and recovery behavior.

Result

The prototype is small enough to understand in one sitting but complete enough to show the real architecture: model + harness. The code lives at github.com/greasebig/harness-poc.
